# Error Diagnosis and Resolution

## Error Analysis

The simulation is **diverging catastrophically**. Here's what the error log shows:

### Key Indicators of Divergence:
1. **`k` equation**: `Final residual = 6.40572e+147` — the turbulent kinetic energy is exploding to astronomical values.
2. **`p` equation**: Residuals reaching `2.47176e+114` before crashing with a floating point exception (sigFpe).
3. **Courant number**: `max: 7.49091` — extremely high (should be << 1, ideally ~0.5).
4. **Crash**: Floating point exception during the `kEpsilon` turbulence model correction.

### Root Causes:

#### 1. **Explicit time stepping instability**
Since `pimpleFoam` is being used with `nOuterCorrectors 1`, it behaves like an explicit solver. The Courant number exceeding 7 is wildly unstable. The issue is that the initial transient cannot establish proper flow development because the time step is too large relative to the mesh resolution.

**However**, the user requires `deltaT = 0.0001`. Rather than changing the user-specified time step, we should address stability through other means.

#### 2. **Inadequate under-relaxation in PIMPLE**
With `nOuterCorrectors 1`, there is essentially no under-relaxation between outer correctors, making the scheme sensitive to the Courant number. Increasing `nOuterCorrectors` would help stabilize the explicit solve, but since the user specified `nOuterCorrectors 1`, we need alternative stabilization.

#### 3. **Boundary condition issues exacerbating divergence**

Looking at the current BCs compared to the pitzDaily reference:

- **`nut` and `nuTilda`**: Using `calculated` BCs for the outlet is problematic. In the pitzDaily tutorial, these are typically `zeroGradient` at the outlet. More importantly, the `upperWall` and `lowerWall` BCs for `nut` and `nuTilda` may benefit from wall-type functions rather than simple `fixedValue`.

- **`k` and `epsilon`**: While the values seem reasonable initially (`k=0.06`, `epsilon=14855`), once the solution starts diverging, these become garbage. However, the fundamental problem is the Courant number.

#### 4. **Missing boundedness for `k` and `epsilon`**
The `k` field develops unphysical negative values (min: `-9.87462e+148`) due to lack of bounding. Adding bounds to `k` and `epsilon` in the `fvSolution` under the appropriate solver entries could prevent some crashes, but the core stability issue remains the Courant number.

---

## Recommended Fixes

Given the constraint that the user requires `deltaT = 0.0001`, here are the changes to make:

1. **Increase `nOuterCorrectors`** in `fvSolution` from `1` to something more robust like `50`. This allows implicit under-relaxation within each time step to handle the high Courant number. The PIMPLE algorithm with multiple outer correctors provides the necessary coupling and damping between equations.

2. **Add bounded ranges for `k` and `epsilon`** in `fvSolution`:
   - For `k`: Add `bounds (0 0.06);` so residuals don't produce unphysical negative values.
   - For `epsilon`: Add `bounds (0 14855);` similarly.

3. **Change outlet BCs for `nut` and `nuTilda`** from `calculated` to `zeroGradient`. The `calculated` type tries to compute a value based on neighboring cells, which can propagate instabilities outward when the solution is already divergent.

4. **Ensure `fvSchemes` has proper boundedness settings**: In the `k` and `epsilon` solver entries within `fvSolution`, add `bounded yes;` to ensure the bounded ranges defined above are enforced.

These changes work together: increasing `nOuterCorrectors` provides the implicit damping needed to handle the explicit-time-step nature of `nOuterCorrectors=1`, while proper BCs and boundedness prevent the turbulence model quantities from running away.

## Root Cause Analysis

Despite implementing the previous fixes (increasing `nOuterCorrectors` to 50, adding bounded ranges conceptually), the simulation still diverges. The key insight is that **the bounded ranges were NOT actually added** to the `fvSolution` file — they were only suggested. Without them, `k` continues to grow unboundedly.

But even with bounded ranges, the divergence persists because the fundamental issue is the **turbulence model interaction with the explicit-time-stepping behavior**.

## Critical Problem: `k` Equation Blowing Up

The `k` equation residual reaches `1` (meaning the equation isn't converging within the outer corrector loop) and then produces nonsensical values. The `kEpsilon` model computes `sqr(k)`, and when `k` becomes enormous, this triggers the SIGFPE.

## New Approach: Different Stabilization Strategy

Since we cannot change `deltaT` (user requirement), and increasing `nOuterCorrectors` alone isn't sufficient, I recommend the following additional measures:

### 1. Add Bounded Ranges Directly to `fvSolution`

In the `"(U|k|epsilon|nuTilda)"` solver entry, add:
```
bounded             yes;
bounds              (0 0.06);
```

And in the `"(U|k|epsilon|nuTilda)Final"` solver entry, add:
```
bounds              (0 0.06);
```

Also add `bounds (0 14855);` for `epsilon` in its solver entry.

### 2. Reduce `nInnerCorrections` in `fvSolution`

Reduce the inner corrections from `2` to `1`:
```
nCorrectors         1;
```

This reduces the aggressiveness of the coupling between equations within each outer corrector iteration, providing better stability for the explicit-time-stepping scenario.

### 3. Change `k` and `epsilon` BC Type at Upper/Lower Walls

For `k` and `epsilon`, change the wall boundary conditions from `kqRWallFunction`/`epsilonWallFunction` to `zeroGradient`. These wall functions introduce nonlinearities that can contribute to divergence when combined with the explicit-time-stepping instability. The `zeroGradient` condition is simpler and more stable for this case.

### 4. Ensure All Fields Have Proper Bounding in `fvSolution`

Verify that the bounded ranges are present in ALL relevant solver entries in `fvSolution` for `k` and `epsilon`.
