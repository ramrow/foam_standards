# Error Diagnosis and Resolution

## Error Analysis

Looking at the error log, the simulation is running but diverging catastrophically:

1. **Massive continuity errors**: `sum local = 126886` — this indicates the mass balance is completely broken
2. **Negative and enormous values**: `bounding epsilon, min: -1.41933e+16` and `bounding k, min: -4.37746e+12`
3. **Negative Courant number**: `Courant Number mean: -4834.62 max: 1.7496e+08` — the max Courant number is extremely high (>> 1), indicating the time step is way too large for stability
4. **Residuals exploding**: `Ux` residual reaches `5.01335e+40`, `Uy` reaches `3.86784e+09`, `Uz` reaches `1.45686e+29`
5. **Floating point exception (sigFpe)**: The simulation crashes due to numerical instability

## Root Cause

The fundamental problem is that **deltaT = 1 second is far too large** for this natural convection case with buoyantFoam. With strong buoyancy-driven flow and relatively low viscosity, the CFL condition is violated almost immediately, leading to exponential divergence.

Looking at the BernardCells tutorial case, the typical deltaT for this kind of simulation is much smaller — on the order of `0.001` to `0.01` seconds, or even smaller depending on the mesh resolution.

## Proposed Fixes

### 1. Reduce deltaT significantly in `system/controlDict`

Change `deltaT` from `1` to something much smaller like `0.001` or `0.0005`. Given the mesh resolution of `(90 10 20)` cells, the smallest cell size is approximately `9/90 = 0.1` m in x, `1/10 = 0.1` m in y, and `2/20 = 0.1` m in z. For natural convection, a good starting deltaT would be based on the square root of (cellSize²/(ν·Gr)), but practically, trying `deltaT 0.001` first.

Update `deltaT` to `0.001` in `controlDict`.

### 2. Increase writeInterval to accommodate the smaller time steps

Since deltaT is now much smaller, writing every 1 second would produce too few output files. Either:
- Set `writeInterval` to `50000` (if deltaT=0.001 and endTime=1000, about 1M time steps, writing every 50 time steps ≈ 20K output files — acceptable), OR
- Keep `writeInterval 50` but understand this will write very infrequently relative to the adaptive time stepping needs.

Actually, reconsider: with `deltaT 0.001`, writing every 50 time steps means writing every 0.05 seconds, which is fine. But the user requirement says "results are written at intervals of every 50 seconds." To honor the user requirement while maintaining stability, we could use adaptive time stepping.

### 3. Use adaptive time stepping in `system/controlDict`

Enable `adjustTimeStep yes;` and set `maxCo 0.5;` (or lower like `0.2`). Then set `deltaT 1;` as the initial time step with `writeInterval 50;` keeping the user's specified write interval. This way the simulation adapts to stable time steps automatically while writing every 50 physical seconds.

Specifically, in `controlDict`:
- Add `adjustTimeStep yes;`
- Change `maxCo` from the default to `0.5;`
- Keep `deltaT 1;` (this becomes the initial time step)
- Keep `writeInterval 50;`

With adaptive time stepping, the initial deltaT of 1s will be quickly reduced to a stable value by the adaptive algorithm.

### 4. Consider reducing relaxation factors in `system/fvSolution`

The current relaxation factors are all `1` (no relaxation). With adaptive time stepping and the PIMPLE algorithm, some under-relaxation helps stability. Add relaxation factors like:

```
relaxationFactors
{
    equations
    {
        U               0.7;
        p_rgh           0.3;
        h               0.7;
        k               0.7;
        epsilon         0.7;
    }
}
```

### Priority Recommendation

The **primary fix** is enabling **adaptive time stepping** (`adjustTimeStep yes;`) with a reasonable `maxCo` (e.g., `0.5`). This allows the simulation to automatically reduce the time step from the user-specified `deltaT 1` down to a stable value, preventing the divergence seen in the log. Without this, even with a smaller deltaT hard-coded, the PIMPLE multi-corrector scheme can still diverge at the beginning.

If adaptive time stepping alone isn't sufficient, combine it with under-relaxation factors as shown above.
