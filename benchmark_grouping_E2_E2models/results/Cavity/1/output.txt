- Attempt 1 failed due to missing `viscosityModel` in `physicalProperties`
- Now we've added `viscosityModel constant;` successfully, but the crash occurs later when computing the effective viscosity

## Root Cause Analysis

The issue is in the `0/k` and `0/nut` boundary conditions combined with the `kEpsilon` model setup. Here's what's happening:

1. In `0/k`: The `movingWall` uses `type fixedValue; value uniform 0;` — this is correct for k (turbulent kinetic energy).
2. In `0/nut`: The `movingWall` uses `type calculated; value uniform 0;` — this is problematic. With `kEpsilon`, `nut` is derived from `Cmu*k^1.5 / epsilon`. When `k=0` everywhere (as initialized), `nut` becomes `0/epsilon = 0`, which is fine. However, the `calculated` type may trigger internal computation paths that lead to instability.

More critically: **The `omega` field is set to `uniform 1`**, but for `kEpsilon`, the solver internally computes `omega` (or relates to it via `epsilon`). Having `omega = 1` while `k = 0` creates inconsistent turbulence quantities. Additionally, looking at the `kEpsilon` coefficients printed in the log, the solver is using the standard constants (`Cmu=0.09`, `C1=1.44`, `C2=1.92`), which suggests it's reading them from somewhere (possibly the `kEpsilonCoeffs` sub-dictionary isn't properly parsed).

## Proposed Solutions

### 1. Fix `0/nut` boundary conditions

Change the `nut` boundary conditions to use `zeroGradient` instead of `calculated`:

```
nut
{
    movingWall
    {
        type            zeroGradient;
    }
    fixedWalls
    {
        type            zeroGradient;
    }
    frontAndBack
    {
        type            empty;
    }
}
```

The `calculated` type forces the solver to recompute `nut` from `k` and `epsilon` at runtime, which with `k=0` everywhere can cause convergence issues right at startup.

### 2. Initialize `k` with a small non-zero value

Set a small initial turbulent kinetic energy to avoid the `0/0` situation:

```
internalField   uniform 0.01;
```

### 3. Check `omega` initialization

For `kEpsilon`, `omega` is typically related to `epsilon` via `epsilon = Cmu * k^1.5 / l`, where `l = Cmu^(0.75) * k / omega`. Setting `omega` to `1` while `k = 0` is inconsistent. Consider initializing `omega` to a reasonable value like `10` (typical for kEpsilon near walls):

```
internalField   uniform 10;
```

### Recommended Combined Approach

Apply **all three fixes together**:
1. Change `0/nut` boundary conditions from `calculated` to `zeroGradient`
2. Set `0/k` internal field to `uniform 0.01` (small non-zero)
3. Set `0/omega` internal field to `uniform 10`

These changes prevent the solver from encountering singularities (`0/0`) during the first PISO iteration when trying to compute `epsilon` and subsequently `nut`, which triggers the SIGFPE.
