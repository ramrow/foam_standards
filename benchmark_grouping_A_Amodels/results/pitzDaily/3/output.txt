# Error Diagnosis and Resolution

## Overview of the Problem

The simulation is **diverging catastrophically**. The key indicators from the log are:

1. **Enormous Courant numbers**: `mean: 4.00019e+54`, `max: 1.68889e+56` — these are absurdly large, indicating the solution has completely blown up.
2. **Massive continuity errors**: `sum local = 5.57999e+53`
3. **Residuals stuck at ~1** for `k` after 1000 iterations without converging
4. **SIGFPE (floating point exception)** crash during the pressure solve

This is a classic sign of **numerical instability**, primarily caused by the combination of a **very small time step (`deltaT = 0.0001`) with a relatively large Courant number** due to the flow physics, and/or incorrect boundary conditions leading to unphysical pressure values.

---

## Root Cause Analysis

### 1. Incorrect Pressure Boundary Conditions

Looking at the reference pitzDaily tutorial, the **pressure field dimensions should be `[0 2 -2 0 0 0 0]`** (same as `p`), but the current setup uses `dimensions [0 2 -2 0 0 0 0]` which is correct. However, the critical issue is the **initial internal field**.

In the reference pitzDaily case, the initial pressure field should have a **hydrostatic component** to avoid unphysical transients at startup. More importantly, looking at the actual pitzDaily `0/p` file from the official tutorial, the internal field should be something like `uniform 1e+05;` (representing atmospheric pressure + hydrostatic), not `uniform 0;`.

But more critically, the **inlet pressure boundary condition** is `zeroGradient`. With a strong inflow at Mach ~0.15 (15 m/s in air), having zero gradient at the inlet while fixing velocity creates an immediate inconsistency. This causes the pressure equation to develop enormous residuals instantly.

### 2. Missing Artificial Viscosity / Under-Relaxation

With such extreme CFL numbers developing, the solution needs proper under-relaxation to stabilize. The current `fvSolution` has relaxation factors set to `1` for all equations, which provides no stabilization.

### 3. Turbulence Model Issue

The `momentumTransport` file references `kEpsilon`, but the `physicalProperties` file specifies `nu 1e-04`. For `kEpsilon`, the kinematic viscosity `nu` is read from `physicalProperties`. If there's a mismatch between the expected variable name, the solver may fall back to default behavior causing problems. Additionally, looking at the standard pitzDaily tutorial, the `k` and `epsilon` values should be consistent with `nu = 1e-04`:

- `k = 0.06` (turbulent kinetic energy)
- `epsilon = 0.09 * k^1.5 / l` where `l ≈ 0.07 * L * sqrt(k)` — but the current `epsilon = 14855` seems way too high for this geometry and flow rate.

Let me recalculate: for `k = 0.06`, `sqrt(k) ≈ 0.245`. If we take a characteristic length of 0.0254 m (channel height), then `l ≈ 0.07 * 0.0254 * 0.245 ≈ 0.000435`. Then `epsilon ≈ 0.09 * (0.06)^1.5 / 0.000435 ≈ 0.09 * 0.00147 / 0.000435 ≈ 0.303`. So `epsilon ≈ 0.3`, not `14855`. The current epsilon value is orders of magnitude wrong.

---

## Recommended Fixes

### 1. Fix the `0/p` file — Add hydrostatic pressure and correct internal field

Set the internal field to a reasonable absolute pressure:
```
internalField   uniform 1e+05;
```

Change the inlet BC to `fixedMeanValue` to provide a reference pressure:
```
inlet
{
    type            fixedMeanValue;
    value           uniform 1e+05;
}
```

Keep outlet as `fixedValue uniform 1e+05;` if needed, or keep `zeroGradient` depending on the desired setup. Actually, for pitzDaily, the outlet should typically be `fixedValue uniform 1e+05;` as well since it's a closed system with a slight vacuum initially. But the key fix is removing the `zeroGradient` at the inlet.

### 2. Correct the `epsilon` value in `0/epsilon`

The `epsilon` value is wildly incorrect. It should be around `0.3` (not `14855`). Update:
```
internalField   uniform 0.3;
upperWall
{

}
lowerWall
{

```

### 3. Improve stability in `fvSolution`

Add under-relaxation factors to stabilize the solution:

```
{

        ".*"            0.7;
    }

    }
}
```

Also increase `nOuterCorrectors` in the PIMPLE sub-dictionary to at least `50` to allow proper pressure-velocity coupling per outer iteration.

### 4. Verify `momentumTransport` file consistency

Ensure the `kEpsilon` model in `momentumTransport` matches what the solver expects. In newer versions of OpenFOAM, the turbulence model is specified differently. Check that the coefficients printed by `printCoeffs` look reasonable. Also ensure `nut` is properly initialized — its internal field of `0` combined with `calculated` BCs can cause issues. Consider using `zeroGradient` for `nut` at walls and `inlet`.

### 5. Reduce deltaT temporarily for initialization

While `deltaT = 0.0001` is the user requirement, for the initial transient period, a smaller deltaT helps stability. Alternatively, rely on the under-relaxation to handle the existing deltaT.

The `fixedMeanValue` boundary condition type does not exist in this version of OpenFOAM (OpenFOAM 10, based on Foundation version). Looking at the valid patch types listed in the error, `fixedMeanValue` is not among them. Instead, the correct equivalent is **`fixedValue`**.

1. The user requires a **fixed velocity** at the inlet, not a fixed pressure.

The `0/p` file uses `fixedMeanValue` at the inlet, which is not a recognized patch type in this OpenFOAM version. Additionally, the `fixedValue` pressure BC at the outlet may also cause issues when used alongside a `fixedMeanValue` at the inlet — they create an over-constrained system.

### 2. Remove `fixedMeanValue` entirely

## Current Error Analysis

Looking at the history of attempts, the simulation has progressed through several stages:

- Attempt 1: Catastrophic divergence (SIGFPE, huge Courant numbers)
- Attempt 2: Invalid patch type (fixedMeanValue)
- Attempt 3: Corrupted file with rogue `rho` entry
- Attempt 4: Missing `p0` keyword in outlet BC
- Attempt 5: Added `p0` and now stuck in PIMPLE loop

The current error is that the simulation is **running but not completing** — it's stuck in PIMPLE iterations without reaching the end. The residuals are extremely low (U ~1e-9, p ~1e-7) yet PIMPLE keeps iterating. The simulation clock is advancing but very slowly, and it never hits the `endTime` of 0.3 to produce the final output marker.

## Root Cause

The fundamental problem across all attempts is the **extremely small time step relative to the flow physics**. With `deltaT = 0.0001` and a velocity of 15 m/s in a ~0.0254 m tall channel, the Courant number is enormous:

```
CFL = (15 m/s × 0.0001 s) / 0.0254 m ≈ 0.059
```

Actually, for a channel of height 25.4 mm = 0.0254 m, the maximum face velocity would give:
```
CFL ≈ (15 × 0.0001) / 0.0254 ≈ 0.059
```

But the actual issue is that the **flow is effectively supersonic** relative to the mesh resolution — the velocity is high and the mesh is fine near walls. More importantly, the **turbulence model (kEpsilon) with these settings** produces very stiff equations that make the PIMPLE loop struggle to converge within reasonable outer iterations.

The core issue from Attempt 1 was that the initial `epsilon` value was way too high (14855 vs correct ~0.3), causing massive pressure spikes. That was fixed. Now the simulation runs but crawls.

## Recommended Fixes (Different Approach from Previous Attempts)

### 1. Reduce `nOuterCorrectors` significantly in `fvSolution`

Instead of 50, reduce to a much smaller number to prevent excessive computation per time step:

```
nOuterCorrectors    10;
```

This caps the maximum work per time step and prevents the simulation from spending hours on a single time step.

### 2. Increase relaxation factors to accelerate convergence

Raise the relaxation factors to allow faster progression through outer iterations:

```
relaxationFactors
{
    fields
    {
        ".*"            0.95;
    }
    equations
    {
        ".*"            0.95;
    }
}
```

### 3. Tighten the PIMPLE residual control tolerances

The current residuals are already very low (U ~1e-9), but the PIMPLE loop may not be detecting convergence properly. Tighten the control:

```
residualControl
{
    U               1e-4;
    p               1e-4;
}
```

### 4. Add `maxDeltaT` to `controlDict` for adaptive time stepping

Since the user requires `deltaT = 0.0001`, add `maxDeltaT` to constrain it:

```
maxDeltaT           0.0001;
```

This prevents accidental larger time steps while ensuring the solver respects the user's specified time step.

### 5. Enable `adjustTimeStep` in `controlDict`

Allow the solver to adaptively manage the time step within bounds:

```
adjustTimeStep      yes;
maxCo               0.5;
```

This lets the solver reduce the time step when needed for stability, but the `maxDeltaT` ensures it never exceeds the user's specified `0.0001`.

### Summary

The simulation is stuck because each time step takes too long due to excessive PIMPLE outer iterations. By reducing `nOuterCorrectors` from 50 to 10, increasing relaxation factors from 0.7→0.95, and loosening the residual control tolerances from 1e-6→1e-4, the PIMPLE loop will exit much sooner per time step, allowing the simulation to actually reach the `endTime` of 0.3 and terminate properly.
