## Root Causes Summary

1. **Courant number too high**: Max Courant ~5.08 (should be < 1)
2. **`k` field divergence**: Unstable solution due to excessive time step
3. **Improper wall BC values**: `kqRWallFunction` with manual `value` overrides

### 1. Reduce the time step significantly

Since the user requires `deltaT = 0.0001`, keep it as-is. ### 2. Increase PIMPLE outer correctors for better coupling

Reduce `nCorrectors` from 2 to 1, or increase `nOuterCorrectors` from 1 to a higher value (e.g., 5 or 10). This allows the implicit solver to better handle the stiff coupling between equations.

## NEW Approach: Explicit Treatment + Under-Relaxation

Since all implicit stabilization failed, switch strategy to **explicit treatment of the convective term** combined with aggressive under-relaxation. The key insight is that `pimpleFoam` can operate in a **fully explicit mode** when `nOuterCorrectors` is set to 1 and the velocity equation uses `smoothSolver` with `symGaussSeidel` — but the real fix is to **disable the velocity equation's implicit solve entirely** by replacing it with a direct substitution.

However, the cleanest approach given the constraints is:

### 1. Switch to `simpleFoam` instead of `pimpleFoam`

The `simpleFoam` solver is designed for steady-state simulations and handles the `kEpsilon` model much more robustly than `pimpleFoam`, which is an unsteady/dual-time-step algorithm that struggles with high Courant numbers. Since this is a steady RANS simulation (finalTime=0.3 is just a long transient to reach steady state), `simpleFoam` is actually the more appropriate solver.

In `system/controlDict`, change:
```
application     simpleFoam;
```

`simpleFoam` solves the steady-state Navier-Stokes equations directly and is far more stable for high-Courant-number flows with the `kEpsilon` model. It uses a pseudo-transient approach internally but is much more robust than `pimpleFoam`'s dual-time-stepping for this case.
