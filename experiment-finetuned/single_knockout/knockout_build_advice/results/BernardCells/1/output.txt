## Root Cause Analysis

Looking at the error stack trace and the persistent nature of the issue, the fundamental problem is that in **OpenFOAM 10 (Foundation version)**, the `buoyantFoam` solver performs hydrostatic initialisation which computes `p_rgh` from `T` and `g`. The error occurs because somewhere in that computation, the solver is trying to subtract a velocity field from a pressure field.

The key insight is that the `0/p_rgh` file has `internalField uniform 0;` (dimensionless), but the hydrostatic initialisation routine in OpenFOAM 10 expects `p_rgh` to have **pressure-like dimensions** (`[0 2 -2 0 0 0 0]`) so that when it computes `p_rgh = p - rho * g * h`, all terms have consistent dimensions. The previous attempts that set `p_rgh` to `1e5` (pressure-like) didn't work because the solver was still trying to subtract a velocity field from it.

However, the `0/p` file also has `internalField uniform 0;` (dimensionless), and the solver may be accessing `p` during the hydrostatic computation. The real issue is that **both `p` and `p_rgh` need to be initialized with pressure-like values** so that the hydrostatic initialisation can perform dimensionally consistent arithmetic.
