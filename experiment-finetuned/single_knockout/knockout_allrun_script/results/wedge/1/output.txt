# Error Analysis and Resolution

## Error 1: `log.rhoCentralFoam` — Patch type 'patch' not constraint type 'empty'

**Error Message:**
```
patch type 'patch' not constraint type 'empty'
```

**Root Cause:**
The `frontAndBack` patch is defined as `type patch` in `blockMeshDict`, but in the `0/T` file, the boundary condition for `frontAndBack` is set to `empty`. The `empty` constraint type is only valid for patches that are defined as `type empty` in the mesh. When a patch is defined as `type patch` in the mesh, OpenFOAM expects the corresponding boundary condition to be a valid physical boundary condition type (like `fixedValue`, `zeroGradient`, `symmetryPlane`, `slip`, etc.), not `empty`.

**Solution:**
Change the `frontAndBack` boundary condition in `0/T` from `empty` to `zeroGradient`. The `zeroGradient` condition is appropriate for a 3D wedge simulation where the front and back faces are computational boundaries (no physical variation across them):

```
frontAndBack
{
    type            zeroGradient;
}
```

## Error 2: `log.reconstructPar` — Missing `decomposeParDict`

**Error Message:**
```
cannot find file ".../system/decomposeParDict"
```

**Root Cause:**
The simulation is being run in parallel (likely via `decomposePar` + `mpirun`), but no `decomposeParDict` file exists in the `system` directory.

**Solution:**
Create a `decomposeParDict` file in the `system` directory. Since this is a single-domain case (only one processor is used), the file should specify `numberOfSubdomains 1`:

```
numberOfSubdomains 1;

method          scotch;

subdomains
{
    wedge
    {
        mode            absolute;
        cellSubset      all;
    }
}
```

These changes are consistent with the user requirement of a 3D wedge geometry with proper boundary conditions and the `rhoCentralFoam` solver. The `empty` constraint type is only valid when the patch is defined as `type empty` in the mesh, which is not the case here.

**Error Message:**
```
patch type 'patch' not constraint type 'empty'
```

**Root Cause:**
The `frontAndBack` patch is defined as `type patch` in `blockMeshDict`, but in the `0/T` file, the boundary condition for `frontAndBack` is set to `empty`. The `empty` constraint type is only valid for patches that are defined as `type empty` in the mesh. When a patch is defined as `type patch` in the mesh, OpenFOAM expects the corresponding boundary condition to be a valid physical boundary condition type (like `fixedValue`, `zeroGradient`, `symmetryPlane`, `slip`, etc.), not `empty`.

**Solution:**
Change the `frontAndBack` boundary condition in `0/T` from `empty` to `zeroGradient`. The `zeroGradient` condition is appropriate for a 3D wedge simulation where the front and back faces are computational boundaries (no physical variation across them):

```

    type            zeroGradient;
}
```
