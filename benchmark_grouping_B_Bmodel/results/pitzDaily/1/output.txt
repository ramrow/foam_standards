# Error Diagnosis and Resolution

## Error Analysis

The simulation is **diverging catastrophically**. Here's what the error log reveals:

### Key Indicators of Divergence:
1. **Massive residuals**: `Ux` initial residual ~0.999, `Uy` ~0.999 — the solver never converges within 1000 iterations.
2. **Blowing up fields**: `epsilon` bounds go from `-4.7e+25` to `2.1e+27`, `k` goes to `5.6e+25`.
3. **Courant number explosion**: `mean: 5.7e+12`, `max: 4.1e+15` — completely unphysical.
4. **Continuity errors**: `sum local = 2.6e+17` — mass conservation is destroyed.
5. **Floating point exception (sigFpe)** during the pressure solve.

### Root Causes:

#### 1. **Pressure equation singularity**
The pressure field has no reference point (`pRefCell`/`pRefValue`). In a **purely incompressible** simulation like pitzDaily, the pressure equation is singular because pressure is only defined up to a constant. Without a reference cell, the GAMG solver cannot solve the pressure equation properly, leading to divergence.

✅ **Fix**: Add `pRefCell 0;` and `pRefValue 0;` in the `PIMPLE` dictionary of `fvSolution`. These are already present in the pitzDaily tutorial and are required even though the dimensions show `[0 2 -2 0 0 0 0]` (incompressible pressure).

#### 2. **Inlet boundary condition for `p`**
The inlet uses `zeroGradient` for pressure, which is correct for a velocity-driven inflow. However, the combination of a strong velocity inlet with no pressure stabilization can cause issues. Consider adding **pressure relaxation** or using `fixedMeanValuePressure` if needed, but the primary fix should be the `pRefCell`.

#### 3. **Turbulence model parameter mismatch**
The `k` and `epsilon` values need to be consistent with the given kinematic viscosity (`nu = 1e-5`) and the inlet velocity of 10 m/s. For a typical turbulent pipe/channel flow with Re ≈ 10000:

- `k ≈ 1.5 * (U * k_length)^2`
- `epsilon ≈ Cμ^0.75 * k^1.5 / length`

However, since the user specifies `nu = 1e-5` and doesn't mention changing `k` or `epsilon`, keep them as-is unless the simulation explicitly requires different values. The current values may be reasonable depending on the mesh scale.

#### 4. **Relaxation factors too aggressive**
With `nOuterCorrectors 1`, there is essentially no under-relaxation between outer PIMPLE correctors, causing oscillations. Increase `nOuterCorrectors` to 2 or more and add proper relaxation factors (e.g., 0.7 for equations, 0.3 for pressure) to stabilize convergence.

#### 5. **Boundary condition consistency for `nut`**
The `upperWall` and `lowerWall` boundaries for `nut` use `nutkWallFunction`, but the `physicalProperties` file does not define `kappa`, `Cmu`, or `sigma_k`. This may cause issues when computing wall functions. Ensure these coefficients exist in `physicalProperties` or switch to `nut` wall functions that don't require additional coefficients.
