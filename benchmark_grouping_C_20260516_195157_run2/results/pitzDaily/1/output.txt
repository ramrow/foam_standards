## Root Causes

### 1. Turbulence Model Issue (Primary Cause)
The `momentumTransport` file specifies `model kEpsilon;` but the `physicalProperties` file sets `nu 1e-5`. In the pitzDaily tutorial, the `momentumTransport` file should include the full RAS sub-dictionary structure:
```
RAS
{
    model           kEpsilon;
    turbulence      on;
    printCoeffs     on;

    mu              nu;
}
```

However, more critically, the **turbulent viscosity ratio** needs to be controlled. The current `fvSolution` uses `symGaussSeidel` smoother for `k` and `epsilon` equations, which is unstable when there are large Courant numbers. It should use `GaussSeidel` instead.

### 2. Pressure Solver Settings Need Improvement
The GAMG solver has `relTol 0.01` and `tolerance 1e-06`. For PIMPLE with multiple outer correctors, tighter settings are needed:
- Reduce `relTol` to `0.001`
- Increase `smoother` iterations or switch to `GaussSeidel` for better stability

### 3. Under-Relaxation is Missing
The PIMPLE algorithm lacks explicit under-relaxation coefficients. Add relaxation factors in `fvSolution`:

```
relaxationFactors
{
    fields
    {
        p       0.3;
        U       0.7;
        k       0.7;
        epsilon 0.7;
    }
    equations
    {
        p       0.3;
    }
}
```

### 4. Boundary Condition Issues
In the `k` and `epsilon` boundary conditions, using `uniform 0.06` and `uniform 14855` without specifying `valueEntry` format may cause problems. Ensure proper entry format:

Add the following to `fvSolution`:
- Set `nOuterCorrectors` to `2` or higher (currently `1`)

## Error Analysis

The simulation still crashes with a **floating point exception (sigFpe)** during the `kEpsilon` turbulence model correction step. Looking at the current log, the situation has improved significantly compared to Attempt 1:

- The `k` equation residual is now well-controlled (~0.04 for `k`, ~0.017 for `epsilon`)
- The pressure residuals are very small (~1e-3 to 1e-7)
- Only **2 PIMPLE iterations** are needed per time step (vs many before)
- But the crash still occurs in `GaussSeidelSmoother::smooth()` for the `k` equation

This suggests the issue is specifically with the **`k` equation solve** in the `kEpsilon` model's `correct()` function. The `GaussSeidel` smoother is being used but the matrix system for the `k` equation becomes singular or ill-conditioned, causing a divide-by-zero (sigFpe).

## Root Cause

The problem is that the **`k` equation** involves division by the turbulent viscosity `mu_t`, and when the flow develops strong gradients near boundaries, the coefficient matrix develops zeros or negative diagonals, causing the Gaussian elimination to fail.

Looking at the current `fvSchemes`, the convection scheme for `k` is `div(phi,k) Gauss limitedLinear 1;`. This is a second-order central differencing scheme which can produce oscillations and make the `k` equation numerically unstable, especially with the relatively high inlet velocity of 10 m/s.

## Recommended Solutions

### 1. Change the convection scheme for `k` and `epsilon` to first-order upwind (more diffusive, more stable)

In `system/fvSchemes`, change:
```
div(phi,k)      Gauss limitedLinear 1;
div(phi,epsilon) Gauss limitedLinear 1;
```
to:
```
div(phi,k)      Gauss upwind;
div(phi,epsilon) Gauss upwind;
```

First-order upwind is less accurate but much more robust for transient simulations where stability is critical.

### 2. Switch from `GaussSeidel` to `GaussSorted` smoother in `fvSolution`

Replace `GaussSeidel` with `GaussSorted` for the `k` and `epsilon` solvers. `GaussSorted` reorders the equations to improve diagonal dominance and avoids some of the convergence issues that lead to sigFpe.

In `system/fvSolution`, change:
```
"(U|k|epsilon|nuTilda)"
{
    solver          smoothSolver;
    smoother        GaussSeidel;
    ...
}
```
to:
```
"(U|k|epsilon|nuTilda)"
{
    solver          smoothSolver;
    smoother        GaussSorted;
    ...
}
```

### 3. Tighten the pressure solver tolerances slightly

Reduce `relTol` for `p` from `0.001` to `0.0001` and reduce `tolerance` from `1e-06` to `1e-07` to improve coupling stability.

### 4. Verify the `momentumTransport` file

Ensure the `constant/momentumTransport` file contains the full RAS sub-dictionary with `mu nu;` — this was added in the previous attempt and should remain.

## Summary
