1. **Courant numbers are very high**: Mean ~1.79, Max ~1.97 — these are far above 1, indicating the simulation is unstable.

## Root Cause

The combination of **very high Courant numbers** (~2) and the explicit density-based solver (`rhoCentralFoam`) leads to instability. With `deltaT = 0.002` and these mesh dimensions, the CFL condition is violated.

Additionally, looking at the current `fvSolution`, the solvers are set to `diagonal` for `(rho|rhoU|rhoE)`, which means no iterative correction is applied. For explicit density-based solvers like `rhoCentralFoam`, the pressure equation involves a `sqrt()` computation that requires positive pressure/density, and without proper coupling between equations, even small instabilities can cause immediate failure.

## Resolution

Since the user specifies `deltaT = 0.002`, we cannot change the user-provided timestep. Instead, we need to enable adaptive time stepping to keep the Courant number under control while still respecting the user's intent:

1. **Set `adjustTimeStep yes;`** in `controlDict` — this allows the solver to automatically reduce the timestep to maintain stability.
2. **Keep `maxCo 1;`** — this caps the Courant number at 1, ensuring stability for the explicit solver.
3. Keep `deltaT 0.002;` as the initial/maximum deltaT — the solver will use this as the starting point and adjust downward as needed.
4. Change `runTimeModifiable false;` to `true;` so that `adjustTimeStep` can actually modify the timestep during runtime.

These changes ensure the simulation remains stable (keeping Courant < 1) while using the user-specified `deltaT = 0.002` as the basis for the time stepping. Without adaptive time stepping, the Courant number will continue to grow beyond what the explicit solver can handle, leading to the same floating-point exception crash seen here.

1. The `reconstruct()` schemes use `vanLeer`/`vanLeerV` which are TVD limiters — these should prevent negative pressures, BUT...
