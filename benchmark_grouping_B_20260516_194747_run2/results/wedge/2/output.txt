# Error Analysis and Resolution

## Error: Casting `wall` to `symmetryPlane`

**Error Message:**
```
Attempt to cast type wall to type symmetryPlane
```

**Diagnosis:**

This error is very telling. The stack trace shows the error originates from the thermophysical properties construction (`hePsiThermo`), specifically when the solver tries to read the `p` field. Here's what happens:

1. `rhoCentralFoam` constructs the thermophysical properties (`hePsiThermo`)
2. During this process, it looks up the boundary condition for patch `bottom` in the `p` field
3. The `p` field has `bottom` set to `symmetryPlane` ✓
4. But the `T` field has `bottom` set to `zeroGradient` ✗
5. When the thermo system tries to reconcile the `bottom` patch across fields, it encounters inconsistency
6. More precisely, when reading `p`, it finds `bottom` is `symmetryPlane`, but internally it sees a `wall` type (likely from a cached/previous lookup or from the mesh topology), and cannot cast `wall` to `symmetryPlane`

The key insight from the previous attempt is that **all scalar fields (`p`, `T`) must use `symmetryPlane` on `bottom`** to be consistent with `U`'s `symmetryPlane` setting. The `zeroGradient` on `T` was the culprit in Attempt 5.

However, looking at the current files provided, the `T` file already has `bottom` set to `symmetryPlane`. This means the error must be coming from a different source — likely the **mesh topology**.

**Re-examining the current `blockMeshDict`:**

The critical issue is the `bottom` patch definition:
```
bottom
{
    type wall;
    faces ()
}
```

An empty face list with `type wall` can cause problems. In some versions of OpenFOAM, an empty `wall` patch can be internally resolved to a `symmetryPlane` expectation conflict. The face that was previously assigned to `bottom` (the y=0 plane) is now unassigned, creating an inconsistent mesh description.

**Proposed Solution:**

In `blockMeshDict`, change the `bottom` patch to use `symmetryPlane` type instead of `wall`, with an empty face list:

```

{
    type symmetryPlane;
    faces ();
}
```

This ensures the patch type is consistent across all fields. The empty face list means no actual boundary faces are assigned to it — the y=0 plane face is already claimed by the `inlet` patch, so this patch simply doesn't exist geometrically but remains available for boundary condition lookups.
