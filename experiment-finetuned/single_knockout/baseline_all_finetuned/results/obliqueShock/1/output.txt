## Root Cause

The issue is in the **reconstruction scheme** for `rho`. The `fvSchemes` file uses:

```
reconstruct(rho) vanLeer;
```

For `rhoCentralFoam`, the density `rho` is reconstructed from the primitive variables. The `vanLeer` limiter can produce non-physical values (negative densities or pressures) near strong shocks if the mesh resolution is insufficient or if the initial conditions are not well-behaved. With only 200 cells across the x-direction and a strong oblique shock, this can cause negative density values, which then cause the `sqrt` to crash when computing the speed of sound.

## Proposed Solution

**In `system/fvSchemes`**, change the reconstruction scheme for `rho` from `vanLeer` to a more robust limiter that is less likely to produce negative values. Use `minmod` instead:

```

```
to:
```
reconstruct(rho) minmod;
```

The `minmod` limiter is more diffusive but more robust near discontinuities and is less likely to produce negative density values that would cause the `sqrt` crash. This is a common fix for `rhoCentralFoam` simulations that crash with SIGFPE.

