This tells us that the `nutWallFunction` boundary condition type is **not recognized**. Looking at the valid patch types listed in the error, there is indeed **no** `nutWallFunction` type available. The closest equivalents are `nutkWallFunction` and `nutURoughWallFunction`, etc.

When `calculated` is used, the field is evaluated at runtime based on the current `k` and `epsilon` values.

## Root Cause

The `0/nut` file contains a complete field definition with boundary conditions. In OpenFOAM 10 with the `kEpsilon` model, the `nut` field is **not** a primitive variable that gets written out and read back — it is computed on-the-fly. Having a `nut` file at all can cause confusion, especially when boundary conditions don't exactly match what the solver expects.

## Recommended Fix

### Remove the `0/nut` file entirely
