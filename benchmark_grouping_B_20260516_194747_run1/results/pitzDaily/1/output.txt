# Error Diagnosis and Resolution

## Error Analysis

The simulation is **diverging catastrophically**. Here's what's happening:

### Key Indicators:
1. **Continuity errors growing exponentially**: `sum local = 9.11886e+50` — these should be small numbers (ideally ~1e-5 or less).
2. **Negative and enormous bounds for k and epsilon**: `min: -2.40705e+77`, `max: 4.04722e+82`
3. **Courant number exploding**: `mean: 7.79e+50`, `max: 1.20e+54`
4. **Floating point exception (SIGFPE)** during the pressure solve.

### Root Cause: **Explicit time stepping instability**

The simulation uses `pimpleFoam` with `nOuterCorrectors 1`, which makes it effectively a **steady-state SIMPLE algorithm** running under transient mode. With such a high velocity (10 m/s) and very fine mesh (the smallest cells are 0.001 scale, with 18×30×1 blocks), the CFL condition is violated from the first time step. The explicit treatment of convective terms combined with the large time step (`deltaT = 0.0001`) causes immediate divergence.

Additionally, there are **inconsistent wall function treatments** across turbulence variables that will cause problems once the flow develops near walls:

- **k** uses `kqRWallFunction` — this is correct for wall functions.
- **epsilon** uses `epsilonWallFunction` — this is also correct.
- But **nut** uses `nutkWallFunction` — this is inconsistent. When using `kEpsilon`, `nut` should use `nutWallFunction`. Similarly, **nuTilda** uses `fixedValue`/`zeroGradient` at walls, which doesn't align with standard wall-function practices for RAS models.

However, the **primary issue** is the numerical instability due to the Courant number blowing up.

## Recommended Fixes

### 1. Enable implicit time stepping in `controlDict`

Change the PIMPLE outer corrector count so that the solver behaves like PISO (implicit in time):

```
PIMPLE
{
    nOuterCorrectors    50;
    nCorrectors         2;
    nNonOrthogonalCorrectors 0;
    pRefCell            0;
    pRefValue           0;
}
```

With many outer correctors and implicit time integration, the simulation will behave like a steady-state solver with adaptive time stepping. Also consider adding:

```
adjustTimeStep      yes;
maxCo               1;
```

This allows the time step to automatically reduce to maintain stability.

### 2. Fix the `nut` boundary condition in `0/nut`

Change the wall boundary condition for `nut` from `nutkWallFunction` to `nutWallFunction`:

```
upperWall
{
    type            nutWallFunction;
    value           uniform 0;
}

lowerWall
{
    type            nutWallFunction;
    value           uniform 0;
}
```

Also change the inlet/outlet from `calculated` to proper initial values consistent with the kEpsilon model:

```
inlet
{
    type            fixedValue;
    value           uniform 0.09;
}
```

(The turbulent viscosity scale can be derived from k and epsilon: ν_t = Cμ·k²/ε ≈ 0.09²/14855 ≈ 5.5e-4, but since nut is initialized to 0 anyway, the key fix is the wall function type.)

### 3. Verify `nuTilda` boundary conditions

The current `nuTilda` uses mixed Dirichlet/Robin conditions at walls (`fixedValue`), which may conflict with the `nutWallFunction` used by other models. For consistency with the `kEpsilon` model, consider making `nuTilda` use `nutWallFunction` at walls too, or ensure its boundary conditions don't create inconsistencies.

### Summary

The **main fix** is switching to an implicitly coupled PIMPLE configuration with many outer correctors and automatic time stepping (`adjustTimeStep yes`). The current setup treats pimpleFoam as a pure transient solver with only 1 outer corrector, which cannot handle the stiff explicit convection at this CFL. Without changing the PIMPLE configuration, any attempt to fix boundary conditions alone will likely fail — the simulation will continue diverging.

## Root Cause

The simulation is essentially performing 10 full PIMPLE outer corrections for every single time step, even though the residuals are already very small. The "End" marker is only written when the time step completes ALL outer corrections AND reaches the end time. Since the solver never exits the outer loop early (due to lack of a convergence criterion trigger), it consumes all 10 corrections per time step and never produces output.

## Recommended Fixes

```
    nCorrectors         2;
    nNonOrthogonalCorrectors 0;

    pRefValue           0;

}
```

This tells PIMPLE to exit the outer loop early once the residuals for U and p drop below 1e-5, which should happen quickly since they're already at ~1e-5 levels.
