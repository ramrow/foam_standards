## Error: Floating Point Exception (SIGFPE) in `sqrt` function

### Diagnosis

The simulation crashes with a **floating point exception (SIGFPE)** in the `sqrt` function. Looking at the log, the Courant numbers are extremely high (max ~3.77), which means the simulation is running with **no time stepping control** (`adjustTimeStep no`) and the time step is too large for stability. The Courant number is calculated as `CFL = |u| * deltaT / h`, and with `deltaT = 0.002` and the mesh resolution, the CFL condition is violated.

The `rhoCentralFoam` solver uses an explicit time integration scheme (Kurganov-Tadmor), which requires the CFL number to be less than 1 for stability. With max Courant numbers around 3.7, the solution becomes unphysical and eventually produces NaN/inf values that trigger the SIGFPE in the `sqrt` operation (likely computing speed of sound from density/pressure).

### Resolution

The key fix is to **enable adaptive time stepping** so that the simulation respects the CFL condition. Change the `controlDict` as follows:

1. **Set `adjustTimeStep yes;`** — This enables automatic time stepping based on the CFL condition.
2. **Add `maxCo 0.5;`** — This sets the Courant number limit to a safe value (0.5 is a common choice for explicit schemes).
3. **Keep `deltaT 0.002;`** as the initial time step (it will be overridden by the adaptive algorithm).

The corrected `controlDict` should have:

```
adjustTimeStep   yes;
maxCo            0.5;
```

This will allow the solver to automatically reduce the time step to maintain stability, preventing the floating point exception.
