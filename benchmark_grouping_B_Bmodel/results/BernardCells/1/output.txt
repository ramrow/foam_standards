```
keyword rhoFinal is undefined in dictionary ".../system/fvSolution/solvers"
```

This means the `buoyantFoam` solver is looking for a solver entry for `rhoFinal` in the `fvSolution/solvers` sub-dictionary, but it doesn't exist there.

In OpenFOAM's `buoyantFoam` solver (which is a density-based solver), the density field `rho` is solved as part of the compressible flow equations. When using PIMPLE with inner correctors, the solver needs to solve for `rho` and therefore looks for a solver entry like `rhoFinal` (and potentially `rho`). Since neither `rho` nor `rhoFinal` is defined in the solvers section of `fvSolution`, the solver crashes.

Looking at the current `fvSolution`, the solvers are defined for:
- `p_rgh`
- `p_rghFinal`
- `"(U|T|e|h|k|epsilon|R)"` (regex pattern)
- `"(U|T|e|h|k|epsilon|R)Final"` (regex pattern)

But there is no entry for `rho` or `rhoFinal`.

Looking at the history of attempts, we've addressed:
- Patch name mismatches ✓ (resolved)

The remaining issue is that with `deltaT = 1` second on a relatively fine mesh (90×10×20), the simulation develops extremely large pressure/velocity transients that exceed what the current solver settings can handle. The `buoyantFoam` solver needs a much smaller initial time step or a more gradual ramp-up.

## Proposed Solutions

### 1. Enable adaptive time stepping in `system/controlDict`

Add a `maxCo` (maximum Courant number) constraint to let the solver automatically reduce the time step:

```
adjustTimeStep      yes;
maxCo               0.5;
maxDeltaT           1;
```

This will keep the Courant number below 0.5, preventing the numerical blowup seen with the fixed `deltaT = 1`.

### 2. Increase `nOuterCorrectors` and improve relaxation in `system/fvSolution`

With adaptive time stepping, use tighter relaxation for better stability:

```
relaxationFactors
{
    fields
    {
        p_rgh         0.3;
        rho           0.3;
    }
    equations
    {
        ".*"            0.5;
    }
}
```

### 3. Initialize with a smaller `deltaT` in `system/controlDict`

Add a `writeFirstTime yes;` entry so the initial fields are written immediately, and set a smaller initial time step. Alternatively, use `deltaT 0.1;` as the base time step with `maxCo 0.2` to be more conservative.

### 4. Add `reconstructPar` capability for post-processing safety

While not causing the crash, adding `reconstructPar` support ensures output files can be properly read after the simulation completes.

## Recommended Priority Fix

**Enable adaptive time stepping in `system/controlDict`:**

- Add `adjustTimeStep yes;`
- Add `maxCo 0.5;`
- Optionally add `maxDeltaT 1;` to cap the maximum time step at the user-specified value

This is the standard approach for `buoyantFoam` with large time steps. The solver will start with very small time steps (effectively `deltaT → 0`) and gradually grow the time step until it reaches the `maxDeltaT = 1` limit, ensuring numerical stability throughout the simulation.
