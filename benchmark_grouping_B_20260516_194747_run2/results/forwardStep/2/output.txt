## Root Causes

### 1. CFL Condition Violation (Primary Issue)
The maximum Courant number is growing rapidly (1.94 → 2.27), indicating the simulation blows up. With `deltaT = 0.002`, the time step is too large for this mesh and flow conditions. Since `adjustTimeStep` is set to `no`, the solver cannot reduce the time step automatically.

### 2. Missing Dissipation in Reconstruction Schemes
The current reconstruction schemes (`vanLeer` for rho, `vanLeerV` for U, `vanLeer` for T) are second-order accurate but without sufficient dissipation, they can become oscillatory and unstable when Courant numbers exceed ~1.

## Proposed Solutions

### Immediate Fixes

1. **Enable adaptive time stepping** in `controlDict`: Set `adjustTimeStep` to `yes` so the solver can reduce the time step to keep the Courant number under control. Keep `maxCo` at 1 (as it is).

2. **Add a limiter to the reconstruction schemes** in `fvSchemes`: Change the interpolation schemes to use a more dissipative (first-order) option for stability. For example, change:
   ```

   reconstruct(T)  vanLeer;
   ```

   to:
   ```
   reconstruct(rho) Minmod;
   reconstruct(U)  MinmodV;
   reconstruct(T)  Minmod;
   ```

- Change `reconstruct(T)` from `vanLeer` to `Minmod`

The `Minmod` limiter is more diffusive than `vanLeer` and will prevent the oscillations that cause the SIGFPE crash, allowing the simulation to run stably even with higher local Courant numbers.

If `Minmod` is not supported, alternatively setting `maxCo` to a lower value like `0.5` would also help stabilize the simulation, but since the user specifies `deltaT = 0.002`, using a more dissipative reconstruction scheme is the better approach.
