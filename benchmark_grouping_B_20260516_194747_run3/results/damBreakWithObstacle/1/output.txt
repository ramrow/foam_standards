This means the pressure field `p` (or `p_rgh`) doesn't exist in the initial conditions when `interFoam` starts, OR the PIMPLE algorithm in `fvSolution` requires explicit reference pressure settings that aren't provided through the usual boundary condition mechanism.

## Root Cause Analysis

The fundamental problem is that **`subsetMesh` should NOT be run before `interFoam`**. The workflow described in the reference case and typical OpenFOAM usage is:
1. Run `blockMesh` — creates the base mesh
2. Run `topoSet` — creates the `obstacleCell` cellSet and `obstacle` cellZone
3. Run `subsetMesh` — removes cells inside the obstacle (this should happen BEFORE `interFoam`)
4. Set initial conditions (`0/` files)
5. Run `interFoam`

If `subsetMesh` ran successfully, it would have removed cells from the mesh, meaning the resulting mesh would have fewer patches — potentially removing `upperWall` entirely. When `interFoam` then tries to read fields on this altered mesh, it looks for `upperWall` and fails.

But since `subsetMesh` keeps failing (likely because `0/p_rgh` isn't present when it runs), the mesh never gets subsetted, and the subsequent `interFoam` run operates on the full mesh. However, the `subsetMesh` job is likely queued independently and keeps failing, blocking the pipeline.

More importantly, the **real showstopper** is the `interFoam` error about `pRefCell`/`pRefPoint`. In compressible solvers like `interFoam`, a reference pressure is required. The PIMPLE algorithm needs to know where the pressure reference is anchored.

## Proposed Fixes

### 1. Add `pRefCell` and `pRefPoint` to `fvSolution` (PIMPLE block)

Add these entries inside the `PIMPLE` dictionary in `system/fvSolution`:

```
PIMPLE
{
    momentumPredictor   no;
    nOuterCorrectors    1;
    nCorrectors         3;
    nNonOrthogonalCorrectors 0;

    pRefCell            0;
    pRefValue           0;
    pRefPoint           (0 0 0);
}
```

This tells OpenFOAM to anchor the pressure field at cell 0 with a reference value of 0, which is the standard approach for `interFoam`.

### 2. Ensure `subsetMesh` runs AFTER `interFoam` completes successfully

The `subsetMesh` command should be run separately after verifying that `interFoam` produces output correctly. Alternatively, restructure the workflow so that `subsetMesh` is only attempted once, after `blockMesh` and `topoSet` have succeeded, and before any solver runs.
