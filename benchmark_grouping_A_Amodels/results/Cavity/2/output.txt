# Error Diagnosis and Resolution

## Error Analysis

The solver crash with a floating point exception (sigFpe) during the `kEpsilon` model's `correctNut()` function indicates that the simulation is encountering division by zero or invalid numerical operations. The key clues are:

1. **Bounding shows min=max=average=0 for both k and epsilon**: This means the kEpsilon model is producing zero turbulent kinetic energy and zero dissipation rate everywhere, which leads to zero viscosity (`nuEff`), causing the division-by-zero crash in the turbulence model's correction step.

2. **The stack trace points to `kEpsilon::correctNut()`**: The solver is trying to compute `nuEff = nu + Cmu * k^2 / epsilon`, but when k and epsilon are both zero, this becomes `0/0` → NaN → sigFpe.

## Root Cause

The simulation quickly drives k and epsilon to zero because:
- The initial/boundary conditions set k and epsilon to zero everywhere.
- With no turbulence generation mechanism (no explicit source terms), the kEpsilon model has nothing to sustain turbulence.
- As the lid moves, shear should generate turbulence, but the current setup doesn't properly initialize or force the turbulence field to develop.

## Recommended Solutions

### Solution 1: Initialize k and epsilon with non-zero values (Primary Fix)

The `k` and `epsilon` fields in the `0/` directory need proper non-zero initial values to seed turbulence. For the kEpsilon model, typical initialization values are:

- **k** (turbulent kinetic energy): Set to a small non-zero value. A common approach for the lid-driven cavity is:
  ```
  k movingWall { type uniformValue; value uniform (0.005 0 0); }
  k fixedWalls { type uniformValue; value uniform (0.005 0 0); }
  k frontAndBack { type empty; }
  ```

- **epsilon** (turbulent dissipation rate): Derived from k and a length scale. For a characteristic length `l = 0.1` (cavity height):
  ```
  epsilon movingWall { type uniformValue; value uniform (0.001 0 0); }
  epsilon fixedWalls { type uniformValue; value uniform (0.001 0 0); }
  epsilon frontAndBack { type empty; }
  ```

Alternatively, use `kqRWallFunction` and `epsilonWallFunction` with appropriate coefficients. The key point is that **both k and epsilon must start with non-zero values** to avoid immediate division-by-zero in the kEpsilon model's viscosity calculation.

### Solution 2: Add a turbulence intensity and length scale based initialization

A more physically meaningful approach:
```
k movingWall { type uniformValue; value uniform (0.00375 0 0); }  // 0.075 * (1^2)
epsilon movingWall { type uniformValue; value uniform (0.009375 0 0); }  // 0.75 * (1^2) / tau, where tau ~ 0.8
```

### Solution 3: Add explicit turbulence source terms (if initialization alone isn't sufficient)

Add a `fvSource` term in `constant/fvSchemes` or `constant/fvSolution` to inject turbulence near the moving lid. However, this is secondary to fixing the initialization.

## Summary

The **primary fix** is to update the `0/k` and `0/epsilon` boundary conditions to use `uniformValue` (or equivalent) with non-zero values for `movingWall` and `fixedWalls` patches. Specifically:

- In `0/k`: Set `value uniform (0.005 0 0)` for both movingWall and fixedWalls.
- In `0/epsilon`: Set `value uniform (0.001 0 0)` for both movingWall and fixedWalls.

These non-zero initial values provide the necessary seed for the kEpsilon turbulence model to generate turbulence from the shear between the moving lid and stationary walls, preventing the division-by-zero crash in `correctNut()`.
