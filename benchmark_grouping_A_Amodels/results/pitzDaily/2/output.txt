- The `nuTilda` field starts at 0 everywhere, meaning initially there is **no turbulent viscosity**. This causes the first few time steps to produce wildly incorrect results, triggering divergence.

### 2. Pressure Solver Settings (`system/fvSolution`)
The GAMG pressure solver has relatively loose convergence criteria (`relTol 0.01`), but the real problem is that the solution keeps diverging before it can converge properly. However, the main fix needs to come from stabilizing the flow field first.

### 3. Time Step Size
While `deltaT = 0.0001` may satisfy the CFL condition for the laminar case, once turbulence develops the effective speed increases dramatically. Combined with the extreme turbulence intensities, even this small time step becomes insufficient after the first few iterations.

## Root Cause Analysis

Comparing against the pitzDaily reference and the user requirements:

### 1. The fundamental problem: **Turbulence model coefficients are wrong**

The user specifies `nu = 1e-05` (kinematic viscosity). But looking at the `k` and `epsilon` values in the current `0/k` and `0/epsilon`:

- `k = 0.0075` → RMS velocity √k ≈ 0.0866 m/s
- `epsilon = 0.84375`

For a standard k-ε model, the turbulent viscosity is `nu_t = Cμ * k² / ε` where `Cμ ≈ 0.09`.

So `nu_t ≈ 0.09 * (0.0075)² / 0.84375 ≈ 0.0000063 m²/s`

But the **kinematic viscosity** `nu = 1e-05 m²/s` is much smaller than `nu_t`. This means the turbulent viscosity dominates, and the effective viscosity is roughly `~6.3e-4 m²/s`.

However, the **real issue** is that the `momentumTransport` file says `model kEpsilon;` but the `physicalProperties` file sets `nu 1e-05`. In some OpenFOAM versions, when `transportModel Newtonian` is used with `kEpsilon`, the solver expects the kinematic viscosity to be derived from the turbulence quantities OR the `nuTilda` field must be consistent with the `nu` value through the relation `nu = nuTilda + Cμ*k²/ε`.

With `k=0.0075` and `epsilon=0.84375`, we get `k²/ε ≈ 0.0000667`, and `Cμ*k²/ε ≈ 6e-6`. So `nuTilda` should be approximately `nu - Cμ*k²/ε ≈ 1e-05 - 6e-6 = 4e-6`. But `nuTilda` is initialized to 0 everywhere!

This mismatch creates negative or wildly incorrect turbulent viscosities at startup, causing immediate divergence.

### 2. **Boundary condition inconsistency for `nuTilda`**

The `nuTilda` BCs use `fixedValue` at walls and `zeroGradient` at outlet/inlet. But for a k-ε model with specified `nu`, the `nuTilda` field should be computed such that `nuTilda = nu - Cμ*k²/ε` everywhere. Simply setting it to 0 or arbitrary values won't work.

### Proposed Solutions

#### 1. Fix the `momentumTransport` file — change the turbulence model
The user requires RANS with PIMPLE, but does not explicitly specify `kEpsilon`. The current `momentumTransport` uses `kEpsilon`. Try switching to `kOmega` (Shear Stress Transport model), which is more robust and doesn't suffer from the same sensitivity to `epsilon` initialization:

```

{

    turbulence      on;
    printCoeffs     on;
}
```

Also update the `0/k` and `0/epsilon` files to match what `kOmega` expects. For `kOmega`, the `epsilon` field is replaced by `omega` (specific dissipation rate). Remove the `epsilon` file entirely and create a `omega` file in `0/` with:
- `dimensions [0 2 -3 0 0 0 0]`
- `internalField uniform 50;` (since ω = ε/(Cμ*k))
- Appropriate wall-function BCs

#### 2. Fix the `nuTilda` initialization
Add a `constant/setFields/` directory with a `nuTilda` file containing the correct initial field. After `blockMesh`, run `setFields` to compute `nuTilda` as `nuTilda = nu - Cμ*k²/ε`. Set it uniformly to approximately `4e-6`.

#### 3. Refine the `fvSolution` settings
