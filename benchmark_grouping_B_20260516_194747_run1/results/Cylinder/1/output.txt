## Error 1: `log.blockMesh` — Block 0 is "inside-out"

**Error Message:**
```
Block hex (0 1 2 3 4 5 6 7) ... is inside-out
```

This means the vertex ordering in the first hex block (and likely others) does not follow the right-hand rule, resulting in a negative volume.

### Root Cause

The issue is with the **vertex ordering** in the `blockMeshDict`. The vertices need to be arranged so that the block has a consistent, positive volume. Looking at the geometry:

- **Block 0**: Vertices 0(-1,-0.5,0), 1(-0.2,-0.5,0), 2(-0.2,0.5,0), 3(-1,0.5,0), and their z-offsets 4-7.
- The problem is that vertices 0→1→2→3 go from x=-1 to x=-0.2 (positive x-direction) but y goes from -0.5 to 0.5 (positive y-direction). For a hex block, the first face (0 1 2 3) should have the **outward normal pointing in the -z direction** (using the right-hand rule). With the current ordering, the block extends in +z, making it inside-out.

Additionally, looking at the full block definition:
```
hex (0 1 2 3 4 5 6 7)
```
The second set of four vertices (4 5 6 7) are at z = -0.1. Vertex 0 is at (x=-1,y=-0.5,z=0) and vertex 4 is at (x=-1,y=-0.5,z=-0.1). So going from vertices 0→4 moves in **-z**. This means the block extends from z=0 down to z=-0.1, and the face (0 1 2 3) at z=0 should have its normal pointing **+z** (downstream). But with vertices ordered as (0 1 2 3), the right-hand rule gives a normal pointing in -z. **This is backwards.**

### Solution

Reverse the vertex ordering in each hex block definition so that the block follows the right-hand rule consistently. Change each block from:

```
hex (i j k l ...) ...
```

to:

```
hex (l k j i ...) ...
```

Specifically:

- Block 0: `hex (7 6 5 4 3 2 1 0)`
- Block 1: `hex (15 14 13 12 11 10 9 8)`
- Block 2: `hex (23 22 21 20 19 18 17 16)`
- Block 3: `hex (31 30 29 28 27 26 25 24)`

Alternatively, you could keep the vertex definitions as-is and only reverse the block vertex order. The key principle is that when looking at the first face (vertices 0,1,2,3), the fourth vertex (3) should be such that the block extends away from that face in the direction where the cross product of edges (1→2) × (0→1) points outward. In this case, the block extends toward decreasing z, so the first face should have vertices ordered so the normal points in +z.

A more robust approach: **reverse the winding order of all blocks**. Change every `hex (...)` entry to have the vertex order reversed (swap the two sets of four vertices and reverse each tetrad).
