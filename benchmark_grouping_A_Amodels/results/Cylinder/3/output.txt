## Error 1: `log.blockMesh` — Block is "inside-out"

**Error Message:**
```
Block hex (0 1 2 3 4 5 6 7) (100 60 1) simpleGrading (1(1) 1(1) 1(1)) is inside-out
```

**Diagnosis:**

The vertex ordering in the hex block definition creates a negative volume, meaning the block is inside-out. This happens because the front and back face vertex orderings don't follow the right-hand rule convention required by OpenFOAM.

Looking at the current vertex definitions:
- **Front face (z=0):** vertices 0=(-1,-0.4,0), 1=(-1,0.4,0), 2=(1,0.4,0), 3=(1,-0.4,0)
- **Back face (z=0.1):** vertices 4=(-1,-0.4,0.1), 5=(-1,0.4,0.1), 6=(1,0.4,0.1), 7=(1,-0.4,0.1)

For a hex block in OpenFOAM, the standard convention is:
- Vertices 0→1→2→3 form one face, and 4→5→6→7 form the opposite face.
- When looking from outside the block toward the interior, vertices should go counterclockwise.
- The second face (back) must have vertices ordered so that the normal points **away** from the first face.

The issue is that the back face vertices (4,5,6,7) are ordered the same way as the front face when viewed from z=-∞. For an inside-out block, the back face needs to be reordered so that the normals point in opposite directions.

**Fix for `blockMeshDict`:** Reverse the vertex ordering of the back face. Change the block definition from:

```
hex (0 1 2 3 4 5 6 7) ...
```

to:

```
hex (0 3 2 1 4 7 6 5) ...
```

This reverses the winding direction of both faces, making them properly oriented relative to each other. Alternatively, another valid fix would be to keep the original vertex positions but swap the front and back face orderings appropriately.

## The Realization

The user requirement specifies an **offset cylinder** simulation. The current `blockMeshDict` defines a simple rectangular block with no cylinder — it just has inlet, outlet, top, bottom, and frontAndBack patches.

However, the boundary condition files (`0/U`, `0/p`) reference a `cylinder` patch and a `defaultFaces` patch. This suggests the user expects a cylindrical obstacle to be modeled in the mesh.

**The solution is to define the cylinder explicitly in the mesh.** The offsetCylinder tutorial does this by defining a circular region that is excluded from the mesh (i.e., the cylinder surface becomes a boundary patch). This requires:

1. Adding arc edges and wire/curved boundary patches in `blockMeshDict`
2. Using `arc` and `circleArc` edge types to carve out the cylinder region

## Proposed Solution

### 1. `system/blockMeshDict` — Implement the offset cylinder geometry

Following the offsetCylinder tutorial pattern, define the cylinder as a hole in the mesh. Add vertices for the cylinder circle and use `arc` edges to define curved boundaries. The domain should be discretized such that the cylinder surface becomes a boundary patch named `cylinder`. The key elements:

- Define all 8 corner vertices as before (0-7)
- Add intermediate vertices along the arcs of the cylinder
- Use `arc` edges to connect vertices along the cylinder circumference
- Define boundary patches: `inlet`, `outlet`, `top`, `bottom`, `cylinder`, `frontAndBack`
- The `cylinder` patch should use `type wall` and consist of the arc edges defining the inner circular surface

The vertex positions should place the cylinder offset from the center. For example, if the cylinder has radius 0.2 centered at origin, the arc vertices would be at positions like `(0.2*cos(theta), 0.2*sin(theta), z)` for theta = 0°, 45°, ..., 360°.
