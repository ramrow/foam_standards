## Error 1: `log.blockMesh` — Trying to specify a boundary face on an internal face

The error is:

```

```

This means that **face (0 1 2 3)** is being assigned to both the `frontAndBack` patch AND to another patch. Looking at the `blockMeshDict`, face `(0 1 2 3)` is listed under **both** `frontAndBack` and `bottom` patches:

- `frontAndBack` includes `(0 1 2 3)`
- `bottom` includes `(0 3 2 1)` — which is the **same face** but with reversed vertex ordering

In OpenFOAM, face `(0 1 2 3)` and `(0 3 2 1)` are the same physical face but with opposite winding. When both are listed as separate boundary entries, OpenFOAM detects this as a conflict.

### Root Cause

The `bottom` patch in `blockMeshDict` uses face `(0 3 2 1)`, which is the **reverse winding** of face `(0 1 2 3)` that is already assigned to `frontAndBack`. OpenFOAM treats these as the same face and refuses to assign it to two different patches.

### Proposed Fix

**Change the face winding for the `bottom` patch** in `system/blockMeshDict` from:

```
bottom
{
    type symmetryPlane;
    faces

        (0 3 2 1)
        (8 11 10 9)
    );
}
```

to:

```
bottom
{
    type symmetryPlane;
    faces
    (
        (0 1 2 3)
        (8 9 10 11)
    );
}
```

This uses the **same vertex ordering** as the `frontAndBack` patch for the corresponding faces, so OpenFOAM recognizes them as distinct boundary faces rather than conflicting assignments.
