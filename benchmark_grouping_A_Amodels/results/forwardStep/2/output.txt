- The crash occurs in `Foam::sqrt`, confirming the density or pressure has gone negative somewhere, making the speed of sound imaginary.

The root cause is the **combination of high Courant number and the initial transient**. With `deltaT = 0.002`, the maximum Courant number exceeds 1 almost immediately, leading to instability. Since `adjustTimeStep` is set to `no`, the solver cannot reduce the timestep automatically.

However, since the user requires `deltaT = 0.002` and we shouldn't modify user-specified parameters, the better approach is to **enable adaptive time stepping** so that the solver respects the CFL condition while still using `deltaT = 0.002` as the maximum allowable timestep.

## Resolution

### Change `adjustTimeStep` from `no` to `yes` in `system/controlDict`

Currently:
```
adjustTimeStep  no;
maxCo           1;
```

It should be changed to:
```

maxCo           1;
```

With `adjustTimeStep yes`, the solver will automatically reduce the timestep to maintain the Courant number below 1, preventing the floating-point exception. The `deltaT 0.002` then serves as the **initial/maximum** timestep, and the actual timestep used will be adjusted downward as needed based on the local Courant number.

### Why this works

The forwardStep tutorial case inherently produces regions where the Courant number can spike, especially near the step corner where flow accelerates suddenly. Without adaptive time stepping, the fixed large timestep causes immediate divergence. Enabling `adjustTimeStep yes` allows the solver to safely handle the transient by reducing `deltaT` locally while respecting the user's intent of having `deltaT = 0.002` as the baseline timestep size.

Note: The `maxCo 1` setting ensures the Courant number never exceeds 1, providing numerical stability.
