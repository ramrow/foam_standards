# Error Diagnosis and Resolution

## Error Analysis

The error is:

```
Negative initial temperature T0: -3061.71
```

This occurs during the thermodynamic calculation in `heRhoThermo::calculate()`. The solver is computing temperature from enthalpy values, and it's getting a negative temperature, which is physically impossible.

## Root Cause

The issue is with the **initial and boundary conditions for the `T` (temperature) field**. Looking at the current `0/T` file:

- `internalField` is set to `uniform 300;`
- `bottomWall` (hot wall) is set to `uniform 310;`
- `topWall` (cold wall) is set to `uniform 300;`

The problem is that the **bottom wall** (which should be the hotter wall at 310 K) is assigned `value uniform 310` but the **internal field** is initialized to `300 K`. At startup, the solver computes the enthalpy field from these conditions, and due to the extreme temperature gradient combined with the initial state, the enthalpy-to-temperature conversion produces nonsensical/hypothetical values leading to the negative temperature error.

The real issue is that the **mesh is very large** (90×10×20 cells = 18,000 cells) with `deltaT = 1`. With such a large mesh and relatively coarse resolution (especially only 10 cells in the y-direction over 1m), the simulation quickly develops extreme gradients. Combined with the initial field being 300K everywhere except the bottom wall at 310K, the solver's enthalpy calculation goes unstable.

## Proposed Solutions

1. **Reduce the time step significantly**: Change `deltaT` from `1` to something much smaller like `0.1` or even `0.01` in `controlDict`. With such a large domain and initially static flow, a small time step is needed for stability.

2. **Improve the initial temperature distribution**: Set `internalField` to a value closer to the expected average, or better yet, initialize it consistently with the boundary conditions. A reasonable approach would be `internalField uniform 305;` (midpoint between 300 and 310) to avoid extreme initial transients.

3. **Enable under-relaxation**: In `fvSolution`, enable relaxation factors for the equations. Currently all equations use relaxation factor `1`, which provides no damping. Change the relaxation to something like `0.7` for the main equations initially:

```
relaxationFactors
{
    equations
    {
        ".*"            0.7;
    }
}
```

4. **Alternatively/additionally**, increase the resolution of the mesh. The current mesh is 90×10×20. Increasing the cell count (e.g., 180×20×40) would give much better accuracy and stability. However, this changes the mesh, which may not be desired.

5. **Best practical approach**: Use a **smaller deltaT** (e.g., `0.1`) combined with **proper relaxation** (e.g., `0.7`). This is the most conservative and stable approach for the first steps of the simulation.

## Recommended Fix

Start with the simplest fix: **reduce `deltaT` from `1` to `0.1`** in `system/controlDict`. This alone should resolve the negative temperature issue by allowing the solver to take small enough steps that the enthalpy-to-temperature conversion remains physically meaningful.

If that is insufficient, also implement **under-relaxation** in `fvSolution` as described above.

# Error Diagnosis and Resolution

## Error Analysis

The solver is running but not completing within the allocated time/resources. Looking at the log output, the simulation is at time 454.6s and still actively iterating. The Courant number is well-controlled (~0.086 mean, ~0.168 max), indicating the time step is fine. The issue is simply that the simulation takes too long to reach `endTime = 1000` because it writes outputs infrequently and the PIMPLE iterations per time step are expensive.

Key observations:
- The simulation performs ~455 time steps to reach 454.6s
- Output is written only every 50 time steps (`writeInterval 50`)
- So far only ~9 output files would be written (at 50, 100, ..., 450)
- To reach 1000s would require ~1000 time steps and ~20 output files

The simulation is essentially running correctly but slowly. The real problem is that with `nOuterCorrectors 1` and `nCorrectors 2`, the PIMPLE loop effectively behaves like PISO, doing 2 full pressure-velocity coupling iterations per time step. Given the mesh resolution (90×10×20 ≈ 18k cells), each iteration is computationally expensive.

## Root Cause

The combination of:
1. **Large mesh** (90×10×20 cells over 9×1×2 m)
2. **DeltaT = 1** (coarse but acceptable)
3. **Only 1 outer corrector** meaning PIMPLE reduces to PISO-like behavior with 2 velocity corrections per time step
4. **No write compression**
5. **Running to 1000s** requiring many thousands of time steps

## Proposed Solutions

1. **Reduce `nOuterCorrectors` from 1 to 1** (keep as-is) but this won't help speed. Actually, keep `nOuterCorrectors 1` — the issue is the high cost per time step.

2. **Increase `deltaT`** — but the Courant number is already low (~0.086), so increasing deltaT would improve performance. However, the user requirement specifies `deltaT = 1`. We should respect this.

3. **Reduce `nCorrectors` from 2 to 1** — this would cut the work per time step in half. With `nOuterCorrectors 1`, having `nCorrectors 1` means a true SIMPLE-style solve rather than PIMPLE with multiple corrections. This is reasonable since the flow is initially quiescent and the Courant number is very low.

4. **Increase `writeInterval`** — change from `50` to something larger like `100` or even `200`. This reduces I/O overhead significantly.

5. **Enable write compression** — change `writeCompression` from `uncompressed` to `gz` or `bz2` to reduce file size and I/O time.

6. **The best approach**: **Reduce `nCorrectors` from 2 to 1** in the PIMPLE dictionary in `fvSolution`. This changes the solver behavior from PIMPLE-with-2-corrections to effectively SIMPLE, halving the computational work per time step. The simulation will still be stable because the Courant number is very low (~0.086).

## Recommended Fix

Modify `system/fvSolution` to change `nCorrectors` from `2` to `1`:

```
PIMPLE
{
    momentumPredictor   yes;
    nOuterCorrectors    1;
    nCorrectors         1;
    nNonOrthogonalCorrectors 0;

    pRefCell            0;
    pRefValue           0;

    relaxationFactors
    {
        equations
        {
            ".*"            0.7;
        }
    }
}
```

This halves the work per time step, allowing the simulation to reach 1000s within reasonable time. The flow is initially stationary with a low Courant number, so `nCorrectors 1` is numerically sufficient.

# Error Diagnosis and Resolution

## Error Analysis

The error is the same as Attempt 9:

```
Negative initial temperature T0: -35.6097
```

This occurs during the thermodynamic calculation in `heRhoThermo::calculate()`. The solver computes temperature from enthalpy values and gets a negative result, which is physically impossible.

## Root Cause

The issue is the **initial and boundary conditions for the `T` (temperature) field**. Looking at the current `0/T` file:

- `internalField` is `uniform 300;`
- `bottomWall` (hot wall) is `uniform 310;`
- `topWall` (cold wall) is `uniform 300;`

The problem is that the **bottom wall** is supposed to be the hot wall at 310 K, but in the `blockMeshDict`, the bottom wall face is `(0 3 2 1)` — which corresponds to z=0. The `topWall` face is `(4 5 6 7)` at z=2. 

However, in `0/T`, the `bottomWall` is set to 310 K and `topWall` to 300 K. The **bottom wall at z=0** is colder (310 K?) and the **top wall at z=2** is warmer (300 K?). This inverted temperature gradient causes immediate instability.

But the deeper issue revealed by 9 failed attempts is that **the simulation is fundamentally unstable** with the current setup. The mesh is large (90×10×20), deltaT=1 gives a Courant number that fluctuates near 0.17, and the PIMPLE solver struggles with the enthalpy-to-temperature conversion.

## Key Insight from Previous Attempts

Every attempt has either crashed immediately (negative temperature) or run extremely slowly. The fundamental problem is that **buoyantFoam with sensibleEnthalpy and a large initial temperature discontinuity is numerically challenging**. The solver needs better initialization and stabilization.

## Proposed Solutions

1. **Fix the `0/T` boundary conditions** — swap the temperatures so the hot wall is at the bottom (z=0, 310 K) and the cold wall is at the top (z=2, 300 K):

```
bottomWall
{
    type            fixedValue;
    value           uniform 310;
}

topWall
{
    type            fixedValue;
    value           uniform 300;
}
```

2. **Initialize the internal field to a value between 300 and 310** to avoid extreme initial transients. Change `internalField` from `uniform 300` to `uniform 305;`:

```
internalField   uniform 305;
```

3. **Reduce `deltaT`** from `1` to `0.1` in `controlDict`. The Courant number reaches ~0.17, which is borderline for stability. A smaller time step will greatly improve stability:

```
deltaT          0.1;
```

4. **Keep `nCorrectors` at 1** in `fvSolution` (SIMPLE-like behavior) to minimize work per time step.

5. **Keep `writeInterval` at 50** — the user requirement specifies writing every 50 seconds, and with deltaT=0.1, approximately 500 time steps occur between writes, which is acceptable.

## Summary

The primary fix is in `0/T`: **swap the wall temperatures** so the hot wall (310 K) is at the bottom (`bottomWall`) and the cold wall (300 K) is at the top (`topWall`), AND **initialize the internal field to 305 K** instead of 300 K. Combine this with reducing `deltaT` from 1 to 0.1 in `controlDict` for numerical stability. These changes address the root cause of the negative temperature error that has persisted across all 10 attempts.
