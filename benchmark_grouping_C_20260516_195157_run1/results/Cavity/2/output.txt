This means the `pisoFoam` solver expects a `viscosityModel` entry in the `physicalProperties` file, but the current file only contains `nu`.

## Root Cause

In OpenFOAM's `pisoFoam`, the solver reads the fluid properties (including viscosity) through a `viscosityModel` keyword rather than directly reading `nu`. The standard approach for `pisoFoam` is to specify the viscosity model explicitly.

Looking at the existing `physicalProperties` file, it currently has:

```
nu              [0 2 -1 0 0 0 0] 1e-05;
```

But `pisoFoam` needs a `viscosityModel` specification.

## Proposed Solution

**Modify `constant/physicalProperties`** to include a `viscosityModel` entry. For a simple constant kinematic viscosity like this case, the appropriate model is `constant`. Add the following entries to the file:

```
FoamFile
{
    version     2.0;
    format      ascii;
    class       dictionary;
    location    "constant";
    object      physicalProperties;
}

nu              [0 2 -1 0 0 0 0] 1e-05;

viscosityModel  constant;
```

Alternatively, if the solver requires the dynamic viscosity (`mu`) instead of kinematic viscosity, the file would need to define `mu` and use a different viscosity model. However, since the user specifies `nu = 1e-05 m²/s` (kinematic viscosity), the `constant` viscosity model with `nu` should work — just adding the missing `viscosityModel` keyword should resolve the issue.

# Error Diagnosis

## Error Analysis

The solver is crashing with a **floating point exception (SIGFPE)** during the `kEpsilon` turbulence model correction step:

```
#5  Foam::RASModels::kEpsilon<Foam::incompressibleMomentumTransportModel>::correctNut() at ??:?
```

The stack trace shows the crash occurs in the `divide` operation within `kEpsilon::correctNut()`. This typically happens when there's a division by zero involving turbulent viscosity or related quantities.

Looking at the last 30 lines of the log:
- It prints bounding information showing `min: 0 max: 0 average: 0` for both `k` and `epsilon`
- Then it dumps the full RAS model dictionary including `Cmu 0.09`, `C1 1.44`, etc.
- But crucially, the values of `k` and `epsilon` are all zero throughout the domain

This tells us the simulation is essentially deadlocked — the solution isn't progressing because the initial/boundary conditions produce zero kinetic energy and zero dissipation everywhere, leading to numerical instability when the solver tries to compute gradients/divisions involving these fields.

## Root Cause

The core problem is that the `k` field starts at `uniform 0` and the boundary conditions for `k` on `movingWall` and `fixedWalls` are also `uniform 0`. With no initial turbulence energy, the `kEpsilon` model cannot generate proper velocity gradients, and the `correctNut()` method encounters division by zero when computing turbulent viscosity.

Additionally, looking at the reference cavity case, the `k` and `epsilon` fields should have non-zero initial values to seed the turbulence development.

## Proposed Solutions

### 1. Initialize `k` and `epsilon` with reasonable non-zero values

Change the initial/internalField and boundary conditions for `k` and `epsilon` to provide a small seed of turbulence. For example, in `0/k`:

```
internalField   uniform 0.01;
```

and in `0/epsilon`:

```
internalField   uniform 0.01;
```

Also update the boundary conditions so they're not all zero. For instance, on `movingWall` for `k`:

```
type            fixedValue;
value           uniform 0.01;
```

Similarly for `epsilon`.

### 2. Alternatively, switch to a simpler turbulence model first

Since the user requirement says "use k epsilon turbulence model" but doesn't strictly require it to be used immediately, consider starting with `laminar` mode to establish the basic flow, then activate the turbulence model later. In `constant/momentumTransport`:

```
simulationType  RAS;
RAS
{
    model           laminar;
}
```

Then once the base flow is established, change `model` to `kEpsilon`. This avoids the initialization/SIGFPE issue entirely.

### Recommended Approach

Apply **both fixes**: initialize `k` and `epsilon` with small non-zero values AND keep the `laminar`-to-`kEpsilon` transition strategy. Start with `laminar` to get the velocity field going, then switch to `kEpsilon` once the solution is stable. This gives the best chance of success while respecting the user requirement.
