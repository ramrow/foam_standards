# Error Diagnosis and Resolution

## Overview of the Problem

The simulation is **diverging catastrophically**. The residuals are extremely high (close to 1.0 for Ux/Uy), the Courant number is astronomically large (~10^12–10^15), and eventually a floating point exception (sigFpe) occurs during the pressure solve.

This is a classic sign of a **numerical instability** caused by inappropriate time stepping relative to the flow physics.

---

## Root Cause Analysis

### 1. Time Step Too Large for the Mesh and Flow Conditions

The mesh has very fine cells near the walls (especially in the x-direction where there are only ~18 cells across a 20.6 mm span). With `deltaT = 0.0001`, the **Courant number is enormous**, leading to immediate divergence.

Looking at the current `fvSchemes`:
```
div(phi,U)      Gauss linearUpwind grad(U);
```

For `pimpleFoam`, the convection scheme should generally be **upwind** (not `linearUpwind`) for stability when using explicit time integration. The `linearUpwind` scheme is less dissipative and can cause oscillations that trigger divergence.

### 2. Pressure Equation Issues from Zero Gradient Inlet BC

The inlet has `zeroGradient` for pressure, which effectively sets `∂p/∂n = 0`. Combined with a strong inflow velocity of 10 m/s, this creates a **backflow problem** — the pressure equation becomes ill-conditioned because the characteristic direction of information propagation changes with the flow field.

### 3. Lack of Under-Relaxation

With `nOuterCorrectors = 1`, PIMPLE behaves like PISO, meaning each time step does exactly one pressure-velocity coupling iteration. Given the large Courant number and complex geometry, this is insufficient for convergence within each time step.

---

## Recommended Fixes

### A. Reduce the Time Step Significantly

Given the mesh resolution and flow speed, the time step needs to be much smaller. Try:

```
deltaT          0.00001;
```

Alternatively, compute the maximum allowable Courant number based on the smallest cell face area. For a first-order upwind scheme, keep `max(CFL) < 1`.

### B. Change the Convection Scheme in `fvSchemes`

Change the `div(phi,U)` discretization to use `upwind` for robustness:

```
div(phi,U)      Gauss upwind;
```

If higher accuracy is needed after stabilization, switch back to `Gauss linearUpwind` once the solution is stable.

### C. Add Proper Relaxation Factors in `fvSolution`

Increase `nOuterCorrectors` to allow proper PIMPLE outer iterations:

```
nOuterCorrectors    50;
```

Add under-relaxation factors to stabilize the solution within each time step:

```
relaxationFactors
{
    fields
    {
        U               0.7;
        p               0.3;
    }
}
```

Keep `nCorrectors = 2` and `nNonOrthogonalCorrectors = 0` as-is.

### D. Improve the Pressure Boundary Condition

Replace `zeroGradient` at the inlet with a more physically consistent BC. For a strong inflow, use:

```
inlet
{
    type            fixedMeanValue;
    value           uniform 0;
}
```

This prevents the pressure from developing unbounded negative values due to backflow.

### E. Ensure Proper Initial Field for `p`

Set the initial internal field of `p` to something reasonable (e.g., `uniform 0` is fine since we have `fixedMeanValue` at the inlet).

### F. Verify Cell Count Appropriateness

The block `(0 3 4 1 8 11 12 9)` uses `(18 30 1)` cells over 20.6 mm × 25.4 mm. This gives very small cells (~1.14mm in x, ~0.85mm in y). Consider increasing the cell count proportionally if the mesh is too fine for the available computational resources, or ensure the time step is correspondingly reduced.

## Root Cause

The `nOuterCorrectors = 50` setting causes excessive computation per time step. Since the residuals drop to convergence levels quickly, the PIMPLE loop should exit early, but with `nOuterCorrectors 50`, it still performs unnecessary iterations.

## Recommended Fix

### 1. Reduce `nOuterCorrectors` in `system/fvSolution`

Change from `50` to a more reasonable value like `10`:

```
nOuterCorrectors    10;
```

This caps the maximum work per time step while still allowing sufficient iterations for transient buildup. Given that residuals are already at 10⁻¹⁰ level, the solver will typically exit well before reaching 10 outer iterations anyway.

### 2. Enable Residual-Based Outer Loop Exit in `system/fvSolution`

Add residual monitoring to allow PIMPLE to exit early when converged. Modify the PIMPLE sub-dictionary in `fvSolution`:

```
PIMPLE
{
    nOuterCorrectors    10;
    nCorrectors         2;
    nNonOrthogonalCorrectors 0;
    pRefCell            0;
    pRefValue           0;

    residualControl
    {
        U               1e-5;
        p               1e-5;
    }
}
```

The `residualControl` sub-dictionary allows PIMPLE to terminate outer iterations early once the residuals fall below the specified thresholds. With current residuals at ~10⁻⁹ to 10⁻¹⁰, the outer loop will exit after just 1-2 iterations instead of running all 10 (or 50) iterations.
